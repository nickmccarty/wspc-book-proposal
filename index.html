About This Book

Foundation models are now commodities. A 7B-parameter open-source model running on consumer hardware is, in most research and synthesis tasks, competitive with models ten times its size — provided the infrastructure surrounding it is well-designed. This book is about that infrastructure: the scaffolding, evaluation loops, memory systems, and orchestration patterns that transform a capable language model into a reliable, auditable, self-improving agent.

Agentic Harness Engineering introduces 30 named architectural patterns organized into seven sections — Inference, Context Engineering, Verification, Orchestration, Security, Observability, and Self-Improvement — each following a canonical structure: intent, motivation, how it works, consequences, and implementation notes with working code. The patterns are drawn from and grounded in the Upskilled harness, an open-source Python and TypeScript pipeline with over 1,500 logged experimental runs.

The book's central empirical claim — that pipeline design matters more than model choice, for a fixed task domain — is established in Part I through controlled experiments and a calibrated six-dimension quality rubric. Part II delivers the pattern catalog, where every entry is traceable to a specific class of failure documented in Chapter 4's failure taxonomy. The patterns are not theoretical: each one has a before-and-after failure rate measured in the reference run set.

"The first book I am aware of that treats the harness — not the model — as the primary engineering artifact. The pattern catalog format makes it immediately actionable."
Sebastian Raschka — Staff Research Engineer, Lightning AI

Topics include: the producer/evaluator/planner model role separation; saturation gating for novelty-filtered web research; dual-backend memory retrieval (ChromaDB + SQLite FTS5); the Wiggum evaluate–revise loop; Git worktrees as agent state management for parallel orchestration; the inference shim as a unified backend abstraction across Ollama, vLLM, and llama.cpp; four-layer security (AST scanning, path sandboxing, injection detection, SSRF prevention); structured telemetry with Chrome Trace export; and the full data flywheel from runs.jsonl to SFT and DPO training datasets with QLoRA fine-tuning.

The companion codebase runs entirely on local hardware with no external API keys required. A minimal deployment needs only 8 GB of VRAM; a standard deployment runs all three model roles simultaneously on a 24 GB consumer GPU.

Pattern Catalog — Highlights

Each of the 30 patterns follows the canonical Fowler-style structure: a single-sentence intent, an "also known as" field, a how-it-works narrative with interaction diagrams, when-to-use and when-not-to-use guidance, a consequences table of benefits and tradeoffs, and implementation notes with working code from the companion repository.

Section A — Inference Patterns
The substrate on which every other pattern depends: routing, role separation, evaluator rotation, and VRAM management.
Inference
A1 · The Inference Shim
Route all LLM calls through a single chat(model, messages) interface, insulating callers from Ollama, vLLM, llama.cpp, and any OpenAI-compatible endpoint.
Inference
A2 · The Model Role Separation
Assign calls to one of three non-overlapping roles — producer, evaluator, planner — and never use the same model instance for two roles within a single run.
Inference
A3 · The Evaluator Pool
Rotate the evaluator model across runs using a deterministic hash of the run ID, converting systematic per-model scoring bias into measurable, diagnosable variance.
Section B — Context Engineering Patterns
What information reaches the synthesis model — the research half of the pipeline and the single largest determinant of output quality.
Context
B1 · The Planner-First
Intercept every task before any search query is issued, producing targeted queries, knowledge gaps, and known facts. Reduces Wiggum revision rounds by 0.8 on average.
Context
B2 · The Novelty Gate
Score each search result batch against the accumulated knowledge state and discard batches that add no new information before they reach the synthesis prompt.
Context
B3 · The Dual-Backend Memory Store
Maintain a persistent observation store backed by ChromaDB (semantic similarity) and SQLite FTS5 (exact keyword overlap), querying both and re-ranking at retrieval time.
Section C — Verification Patterns
How output quality is measured, diagnosed, and improved after synthesis — including the empirical comparison between ReAct and Wiggum evaluation loops.
Verification
C1 · The Wiggum Loop
A separate evaluator model scores output on six dimensions; dimensional feedback routes to the producer for targeted revision; repeats up to three rounds. Raises mean score from 6.87 to 8.12.
Verification
C2 · The Dimensional Rubric
Six named dimensions — Relevance, Completeness, Depth, Specificity, Structure, Groundedness — scored at 0.5-point resolution. The dimensional breakdown, not the composite, drives revision.
Verification
C4 · The ReAct Comparator
Decision criteria for choosing between ReAct evaluation loops and the Wiggum Loop, with empirical data on where each outperforms the other across task type and model availability.
Section D — Orchestration Patterns
Coordinating multiple agents, routing tasks across instances, and extending the platform with slash-command skills.
Orchestration
D1 · The DAG Orchestrator
Decompose broad tasks into a dependency graph of subtasks, verify acyclicity with Kahn's algorithm, execute independent subtasks concurrently, and assemble a cross-reference synthesis.
Orchestration · Sample Chapter
D2 · The Worktree Context
Provision each concurrent subtask with an isolated, checkpointable Git worktree before dispatch — eliminating shared-state race conditions without container overhead. Closes with the coordination problem that isolation cannot solve.
Orchestration
D3 · The MCP Dispatch Router
Route subtasks to remote harness instances over HTTP using keyword-pattern matching, enabling cross-team orchestration without modifying the local execution model.
Section E — Security Patterns
Four defense layers implemented entirely in the Python standard library — no external security dependencies required.
Security
E1 · The AST Guard
Analyze agent-generated Python at the syntax-tree level before execution, blocking dangerous constructs (subprocess calls, dynamic eval, out-of-sandbox writes) without sandboxing infrastructure.
Security
E3 · The Injection Scanner
Detect prompt injection in external content before it reaches the memory store — warn severity for synthesis, block severity for memory writes — with asymmetric enforcement that reflects asymmetric risk.
Security
E4 · The CDP Guard
Prevent the browser skill from navigating to localhost, private IP ranges, or file:// paths, blocking SSRF attacks where a malicious page instructs the agent to fetch internal services.
Section G — Self-Improvement Patterns
Closing the loop between pipeline outputs and model training — every run contributes to the next model generation.
Self-Improvement
G1 · The Data Flywheel
Extract SFT and DPO datasets from runs.jsonl, using Wiggum scores as quality labels and PASS/FAIL pairs as preference signals — no human annotation pipeline required to start.
Self-Improvement
G2 · The RL Rollout
Use the Wiggum Loop as an online reward signal during training — generating candidates, scoring with the rubric, and packaging preference pairs for QLoRA fine-tuning without a separate reward model.
Self-Improvement

Chapter 14: The Worktree Context
Git Worktrees as Agent State Management

The first production implementation of parallel agent orchestration in this harness ran 43 logged tasks before a problem was detected — and only then because a spot-check revealed a discrepancy at run 61. Eighteen runs had passed evaluation with a hidden defect.

The failure is this: when two research agents working on related subtasks both derive their output filenames from a hash of the task string, the filenames collide. The second writer overwrites the first. The run log shows two completions. The filesystem shows one file.

The assembly model reads the available outputs and produces a document that looks, on the surface, like a complete multi-perspective synthesis. It is not. The evaluator scores it, and it passes — because the evaluator cannot detect the absence of content it never saw. The run goes into runs.jsonl as a PASS.

There is no crash. There is no warning. There is no indication that anything went wrong.

This is one of the few failure modes in the architecture that does not surface through the quality evaluation system, because the evaluator can only assess what is present, not what is missing. That makes it the most dangerous kind: silent, plausible-looking, and repeatable across every subsequent parallel run until the root cause is diagnosed.

The fix is forty lines of code. The problem is that you have to know to look for it. This chapter describes the pattern that makes the failure structurally impossible.

The Pattern
Intent
Use Git worktrees to provide each concurrent subtask with an isolated, checkpointable, git-native filesystem environment during multi-agent orchestration, eliminating shared-state race conditions without introducing container overhead or abandoning the repository's history.
Also known as
Worktree-per-Agent; Branched Execution Contexts; Git-Native State Isolation.
Applicability
Use when two or more subtasks run concurrently and write to paths that could collide; when long-running subtasks need to checkpoint progress; when post-run debugging requires reconstructing exactly what any individual subtask wrote. Do not use when all subtasks write to disjoint, statically known paths — simple path-prefixing is sufficient and cheaper.
What Is a Git Worktree

A git clone creates a full copy of the repository. A git worktree add creates a linked working tree. The .git directory — the object store, the refs, the configuration — is shared. The working tree and the index are separate. Two subtasks running in different worktrees literally cannot touch the same files during execution, but after execution their outputs are in the same repository's history.

project-root/ ├── .git/ ← shared by all worktrees │ ├── objects/ ← one object store │ ├── refs/ ← all branches visible to all worktrees │ └── worktrees/ ← git's internal record of linked worktrees ├── harness/ ← main worktree (HEAD branch) └── .worktrees/ ← where linked worktrees are provisioned ├── subtask-a7f2/ ← its own working tree, on branch subtask/a7f2 └── subtask-b3e1/ ← isolated; writes here never affect a7f2
# Create an isolated worktree on a new branch
git worktree add -b subtask/a7f2 .worktrees/subtask-a7f2 HEAD

# Remove it after use (--force handles uncommitted changes)
git worktree remove --force .worktrees/subtask-a7f2
git branch -d subtask/a7f2
git worktree prune
Structure
Orchestrator │ ├── _check_dag_cycles(subtasks) -- Kahn's algorithm; abort if cycles │ ├── _create_worktree(path, branch) -- one per subtask, before dispatch │ ├── ThreadPoolExecutor(max_workers=4) │ ├── _execute_subtask(s1, wt1, …) ─┐ │ ├── _execute_subtask(s2, wt2, …) ├── concurrent; isolated by worktree │ ├── _execute_subtask(s3, wt3, …) │ │ └── _execute_subtask(s4, wt4, …) ─┘ │ ├── collect outputs from each worktree │ ├── _remove_worktree(path) -- cleanup, in finally block │ └── _assemble(task, outputs, model) -- cross-reference synthesis
Implementation
Orchestrator Entry Point

The finally block is the most important implementation detail in the entire function. Worktree cleanup must happen regardless of what else goes wrong. An uncleaned worktree leaves an orphaned branch and a locked working tree that git will refuse to check out again. Over enough failed runs, orphaned worktrees accumulate and consume disk space.

# harness/orchestrator.py

def orchestrate(task: str, output_path: str, **kwargs) -> str:
    memory_context = memory.get_context(task)
    plan = planner.make_plan(task, memory_context)
    subtasks = plan.subtasks

    if not subtasks:
        return agent.run(task, output_path, **kwargs)

    _check_dag_cycles(subtasks)

    # Phase 1: provision worktrees before any subtask starts
    worktrees: dict[str, Path] = {}
    for st in subtasks:
        branch  = f"subtask/{st.id}"
        wt_path = WORKTREE_BASE / st.id
        _create_worktree(wt_path, branch)
        worktrees[st.id] = wt_path

    # Phase 2: concurrent execution
    outputs: dict[str, str] = {}
    errors:  dict[str, str] = {}

    try:
        with ThreadPoolExecutor(max_workers=SUBTASK_MAX_WORKERS) as pool:
            futures = {
                pool.submit(_execute_subtask, subtask=st,
                            worktree=worktrees[st.id], kwargs=kwargs): st
                for st in subtasks
            }
            for future in as_completed(futures):
                st = futures[future]
                try:
                    outputs[st.id] = future.result()
                except Exception as exc:
                    errors[st.id] = str(exc)
    finally:
        # Phase 3: cleanup — always runs, even on KeyboardInterrupt
        for st_id, wt_path in worktrees.items():
            _remove_worktree(wt_path)
            _delete_branch(f"subtask/{st_id}")
        _prune_worktrees()

    # Phase 4: assembly
    return _assemble(task, outputs, model=kwargs.get("model"), errors=errors)
Subtask Execution

The cwd=str(worktree) argument is what makes the isolation real. Every file operation in agent.run() — reading the memory database, writing the output, appending to the research cache — resolves relative to the worktree root, not the main working directory.

def _execute_subtask(subtask: Subtask, worktree: Path, kwargs: dict) -> str:
    output_path = worktree / "data" / "eval" / f"{subtask.id}.md"
    output_path.parent.mkdir(parents=True, exist_ok=True)

    content = agent.run(
        task=subtask.task,
        output_path=str(output_path),
        cwd=str(worktree),   # <-- the key parameter
        **kwargs,
    )

    if os.getenv("HARNESS_CHECKPOINT_SUBTASKS"):
        _checkpoint(worktree, subtask)

    return content
Cycle Detection — Kahn's Algorithm

Before any worktree is provisioned, the orchestrator verifies that the subtask dependency graph has no cycles. A cycle — subtask A depends on B, B depends on C, C depends on A — would cause a deadlock in the thread pool. The check is linear in the number of subtasks and runs before any worktree is created, so a bad decomposition fails immediately.

def _check_dag_cycles(subtasks: list[Subtask]) -> None:
    in_degree = {st.id: 0 for st in subtasks}
    adj: dict[str, list[str]] = {st.id: [] for st in subtasks}

    for st in subtasks:
        for dep in st.depends_on:
            adj[dep].append(st.id)
            in_degree[st.id] += 1

    queue = [sid for sid, deg in in_degree.items() if deg == 0]
    visited = 0

    while queue:
        node = queue.pop(0)
        visited += 1
        for neighbor in adj[node]:
            in_degree[neighbor] -= 1
            if in_degree[neighbor] == 0:
                queue.append(neighbor)

    if visited != len(subtasks):
        raise OrchestratorError("Subtask dependency graph contains a cycle.")
What Worktrees Cannot Do

The pattern solves filesystem isolation completely. It does not solve coordination.

Consider a twelve-subtask orchestration where four subtasks run concurrently. Subtask A3 surveys inference backends; subtask A7 analyzes benchmark results for those same backends. A3 finishes first and concludes that llama.cpp is the optimal backend. A7 is now running in its own worktree, unaware of A3's conclusion — and retrieves benchmark data that either duplicates A3's effort or, worse, reaches a contradictory conclusion based on different sources.

Neither finding is wrong. They were derived from different search rounds, different sources, different search queries. The assembly model will synthesize them and produce a reasonable final document. But A7 spent eighteen minutes on a question that A3's output could have partially answered.

The worktree pattern provides isolation. It provides no mechanism for running subtasks to communicate intermediate findings to each other. For that, you need a message-passing layer — a shared channel that completed subtasks can publish to, and running subtasks can query. In the harness architecture, this is the role of the MCP server. But using the MCP server for mid-execution coordination means subtasks are no longer isolated: the order of completion now affects the content of later outputs.

This is not a failure of the worktree pattern. It is the next problem up the stack, and it requires a different solution.

Consequences

What You Gain

  • Deterministic isolation. Two subtasks writing to the same logical path write to different physical files. No race conditions on output files, memory databases, or research caches.
  • Checkpointability. Long-running subtasks can commit progress to their branch at any granularity. Interrupted runs leave inspectable branches in the repository.
  • Auditability. Even after cleanup, runs.jsonl records each subtask's branch name and commit hash. Exact subtask state is reconstructible.
  • Zero additional dependencies. Every harness deployment is already a git repository. No container runtime, no virtual filesystem, no message broker.

What You Give Up

  • Disk space. Each worktree duplicates the working tree. A 500 MB repo with eight concurrent subtasks temporarily consumes 4 GB of additional disk.
  • Branch hygiene. Subtask branches accumulate. The finally block handles normal completions; interrupted runs leave orphaned branches requiring periodic git worktree prune.
  • Coordination transparency. Because subtasks are isolated, the final document's coherence depends entirely on the assembly model. The quality evaluation system cannot catch contradictions that arise from independent subtasks that did not share information.

Summary
Git worktrees provide isolated, checkpointable, git-native working environments for concurrent subtasks without container overhead or process isolation. The pattern's implementation is forty lines of subprocess calls in orchestrator.py. Its finally block is the most important production engineering decision in the orchestrator. The pattern solves the shared-state problem completely for filesystem isolation. It does not solve coordination — and that limitation is not a defect to be engineered away. It is the boundary between two distinct architectural problems, each with its own solution.
The next chapter examines what happens when the harness escapes the boundary of a single machine entirely: when subtasks are not parallel threads in the same process, but distributed agents running on different hosts, each with different models, different memory stores, and different security boundaries — and the orchestrator must coordinate them over HTTP using the same Model Context Protocol that allows Claude Code to treat the harness as a tool.

At that scale, filesystem isolation becomes the least of your problems.
