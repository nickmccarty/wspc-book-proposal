Chapter 14: The Worktree Context

Git Worktrees as Agent State Management
Agentic Harness Engineering Nicholas McCarty World Scientific, 2026 Pages 187–214

The first time you run five research agents in parallel without worrying about state isolation, the failure mode is so mundane it takes a moment to understand what happened.

Three of the agents finish close together. They each call write_output(), each targeting a filename derived from the task string, and since they are all variants of the same parent task, two of them hash to the same path. The last writer wins. The third agent's output — eighteen minutes of inference, three Wiggum evaluation rounds, a final score of 8.4 — is silently overwritten. The run log shows three completions. The filesystem shows one file.

You do not notice immediately. The assembly model reads the available outputs and synthesizes them into a cross-reference document that looks, on the surface, like a complete multi-perspective analysis. It is not. It is a two-perspective analysis presented in a three-perspective frame, with the third slot filled by the assembly model's own synthesis of the two it could see. The output passes Wiggum evaluation. It gets a 8.2. It goes into runs.jsonl as a PASS.

This is not a crash. There is no stack trace. The pipeline reports success and it is wrong.

Shared mutable state in concurrent agent execution is one of the few failure modes in this architecture that does not surface through the quality evaluation system, because the evaluator cannot know that content is missing — only that what is present is adequate. That makes it the most dangerous kind of failure: silent, plausible-looking, and repeatable.

This chapter describes the pattern that fixes it.

The Pattern

Intent
Use Git worktrees to provide each concurrent subtask with an isolated, checkpointable, git-native filesystem environment during multi-agent orchestration, eliminating shared-state race conditions without introducing container overhead or abandoning the repository's history.
Also known as
Worktree-per-Agent; Branched Execution Contexts; Git-Native State Isolation.
Applicability
Use when two or more subtasks run concurrently and write to paths that could collide; when long-running subtasks need to checkpoint progress; when post-run debugging requires reconstructing exactly what any individual subtask wrote. Do not use when all subtasks write to disjoint, statically known paths — simple path-prefixing is sufficient and cheaper.

What Is a Git Worktree

A git clone creates a full copy of the repository. A git worktree add creates a linked working tree. The .git directory — the object store, the refs, the configuration — is shared. The working tree and the index are separate. Two subtasks running in different worktrees literally cannot touch the same files during execution, but after execution their outputs are in the same repository's history.

project-root/ ├── .git/ ← shared by all worktrees │ ├── objects/ ← one object store │ ├── refs/ ← all branches visible to all worktrees │ └── worktrees/ ← git's internal record of linked worktrees ├── harness/ ← main worktree (HEAD branch) └── .worktrees/ ← where linked worktrees are provisioned ├── subtask-a7f2/ ← its own working tree, on branch subtask/a7f2 └── subtask-b3e1/ ← isolated; writes here never affect a7f2
# Create an isolated worktree on a new branch
git worktree add -b subtask/a7f2 .worktrees/subtask-a7f2 HEAD

# Remove it after use (--force handles uncommitted changes)
git worktree remove --force .worktrees/subtask-a7f2
git branch -d subtask/a7f2
git worktree prune

Structure

Orchestrator │ ├── _check_dag_cycles(subtasks) -- Kahn's algorithm; abort if cycles │ ├── _create_worktree(path, branch) -- one per subtask, before dispatch │ ├── ThreadPoolExecutor(max_workers=4) │ ├── _execute_subtask(s1, wt1, …) ─┐ │ ├── _execute_subtask(s2, wt2, …) ├── concurrent; isolated by worktree │ ├── _execute_subtask(s3, wt3, …) │ │ └── _execute_subtask(s4, wt4, …) ─┘ │ ├── collect outputs from each worktree │ ├── _remove_worktree(path) -- cleanup, in finally block │ └── _assemble(task, outputs, model) -- cross-reference synthesis

Implementation

Orchestrator Entry Point

The finally block is the most important implementation detail in the entire function. Worktree cleanup must happen regardless of what else goes wrong. An uncleaned worktree leaves an orphaned branch and a locked working tree that git will refuse to check out again.

# harness/orchestrator.py

def orchestrate(task: str, output_path: str, **kwargs) -> str:
    memory_context = memory.get_context(task)
    plan = planner.make_plan(task, memory_context)
    subtasks = plan.subtasks

    if not subtasks:
        return agent.run(task, output_path, **kwargs)

    _check_dag_cycles(subtasks)

    # Phase 1: provision worktrees before any subtask starts
    worktrees: dict[str, Path] = {}
    for st in subtasks:
        branch  = f"subtask/{st.id}"
        wt_path = WORKTREE_BASE / st.id
        _create_worktree(wt_path, branch)
        worktrees[st.id] = wt_path

    outputs: dict[str, str] = {}
    errors:  dict[str, str] = {}

    try:
        with ThreadPoolExecutor(max_workers=SUBTASK_MAX_WORKERS) as pool:
            futures = {
                pool.submit(_execute_subtask, subtask=st,
                            worktree=worktrees[st.id], kwargs=kwargs): st
                for st in subtasks
            }
            for future in as_completed(futures):
                st = futures[future]
                try:
                    outputs[st.id] = future.result()
                except Exception as exc:
                    errors[st.id] = str(exc)
    finally:
        # Phase 3: cleanup — always runs, even on KeyboardInterrupt
        for st_id, wt_path in worktrees.items():
            _remove_worktree(wt_path)
            _delete_branch(f"subtask/{st_id}")
        _prune_worktrees()

    return _assemble(task, outputs, model=kwargs.get("model"), errors=errors)

Subtask Execution

The cwd=str(worktree) argument is what makes the isolation real. Every file operation in agent.run() — reading the memory database, writing the output, appending to the research cache — resolves relative to the worktree root, not the main working directory.

def _execute_subtask(subtask: Subtask, worktree: Path, kwargs: dict) -> str:
    output_path = worktree / "data" / "eval" / f"{subtask.id}.md"
    output_path.parent.mkdir(parents=True, exist_ok=True)

    content = agent.run(
        task=subtask.task,
        output_path=str(output_path),
        cwd=str(worktree),   # <-- the key parameter
        **kwargs,
    )

    if os.getenv("HARNESS_CHECKPOINT_SUBTASKS"):
        _checkpoint(worktree, subtask)

    return content

Cycle Detection — Kahn's Algorithm

Before any worktree is provisioned, the orchestrator verifies that the subtask dependency graph has no cycles. A cycle would cause a deadlock in the thread pool. The check is linear in the number of subtasks and runs before any worktree is created, so a bad decomposition fails immediately rather than partway through a multi-minute execution.

def _check_dag_cycles(subtasks: list[Subtask]) -> None:
    in_degree = {st.id: 0 for st in subtasks}
    adj: dict[str, list[str]] = {st.id: [] for st in subtasks}

    for st in subtasks:
        for dep in st.depends_on:
            adj[dep].append(st.id)
            in_degree[st.id] += 1

    queue = [sid for sid, deg in in_degree.items() if deg == 0]
    visited = 0

    while queue:
        node = queue.pop(0)
        visited += 1
        for neighbor in adj[node]:
            in_degree[neighbor] -= 1
            if in_degree[neighbor] == 0:
                queue.append(neighbor)

    if visited != len(subtasks):
        raise OrchestratorError("Subtask dependency graph contains a cycle.")

What Worktrees Cannot Do

The pattern solves filesystem isolation completely. It does not solve coordination.

Consider a twelve-subtask orchestration where four subtasks run concurrently. Subtask A3 surveys inference backends; subtask A7 analyzes benchmark results for those same backends. A3 finishes first and concludes that llama.cpp is the optimal backend. A7 is now running in its own worktree, unaware of A3's conclusion — and retrieves benchmark data that either duplicates A3's effort or, worse, reaches a contradictory conclusion based on different sources.

Neither finding is wrong. They were derived from different search rounds, different sources, different queries. The assembly model will synthesize them and produce a reasonable final document. But A7 spent eighteen minutes on a question that A3's output could have partially answered.

The worktree pattern provides isolation. It provides no mechanism for running subtasks to communicate intermediate findings to each other. For that, you need a message-passing layer — a shared channel that completed subtasks can publish to, and running subtasks can query. In the harness architecture, this is the role of the MCP server. But using the MCP server for mid-execution coordination means subtasks are no longer isolated: the order of completion now affects the content of later outputs.

This is not a failure of the worktree pattern. It is the next problem up the stack, and it requires a different solution.

Mathematical Note — The Isolation Property and the Coordination Gap

Let \(S = \{s_1, s_2, \ldots, s_n\}\) be the set of concurrent subtasks dispatched by the orchestrator, and let \(\mathcal{W}(s_i)\) denote the working tree provisioned for subtask \(s_i\). The isolation property guaranteed by the Worktree Context is:

\[ \mathcal{W}(s_i) \cap \mathcal{W}(s_j) = \emptyset \quad \forall\, i \neq j \]

This eliminates all shared-state race conditions: no write in \(\mathcal{W}(s_i)\) can affect any read in \(\mathcal{W}(s_j)\). The property is exact — it holds for output files, memory databases, and research caches alike, because all file operations resolve relative to the worktree root.

The isolation property is, however, precisely the source of the coordination gap. Define the semantic overlap between two subtasks as the mutual information between their task descriptions:

\[ \Omega(s_i, s_j) = I\!\left(\,\text{desc}(s_i)\,;\,\text{desc}(s_j)\,\right) \]

Under worktree isolation, the information available to \(s_j\) during execution is independent of any output produced by \(s_i\), regardless of \(\Omega\):

\[ I\!\left(\,\text{out}(s_i)\,;\,\text{process}(s_j)\,\right) = 0 \quad \text{even when } \Omega(s_i, s_j) > 0 \]

When \(s_i\) completes before \(s_j\) and \(\Omega > 0\), there is an unrealized informational opportunity cost — the reduction in \(s_j\)'s uncertainty about its task that \(s_i\)'s output could have provided, but cannot:

\[ C(s_j \mid s_i) = H\!\left(\text{desc}(s_j)\right) - H\!\left(\text{desc}(s_j) \mid \text{out}(s_i)\right) \geq 0 \]

In the A3/A7 example: \(\Omega > 0\) (both subtasks concern inference backend selection), A3 completes at \(t = 12\), and A7 runs until \(t = 18\) without access to A3's conclusion. The six wasted minutes are a direct consequence of \(C(s_{A7} \mid s_{A3}) > 0\) combined with zero information flow.

The cycle detection check (Kahn's algorithm, §Implementation) runs in \(O(V + E)\) time, where \(V = |S|\) is the number of subtasks and \(E\) is the number of dependency edges in the DAG. For the typical harness workload (\(|S| \leq 16\), sparse \(E\)), this is negligible relative to inference time, which justifies running it eagerly before any worktree is provisioned.

Finally, note that the assembly model receives \(\{\,\text{out}(s_i)\,\}_{i=1}^n\) without knowledge of \(\Omega\). Its synthesis quality is bounded by its ability to detect and resolve contradictions that arise when \(\Omega(s_i, s_j) > 0\) and the outputs disagree — a capability that is a function of assembly prompt design (Section F, Observability) rather than the orchestration pattern itself.

Consequences

What You Gain

  • Deterministic isolation. Two subtasks writing to the same logical path write to different physical files. No race conditions on output files, memory databases, or research caches.
  • Checkpointability. Long-running subtasks can commit progress to their branch at any granularity. Interrupted runs leave inspectable branches.
  • Auditability. Even after cleanup, runs.jsonl records each subtask's branch name and commit hash. Exact subtask state is reconstructible.
  • Zero additional dependencies. Every harness deployment is already a git repository. No container runtime, no virtual filesystem, no message broker.

What You Give Up

  • Disk space. Each worktree duplicates the working tree. A 500 MB repo with eight concurrent subtasks temporarily consumes 4 GB of additional disk.
  • Branch hygiene. Subtask branches accumulate. The finally block handles normal completions; interrupted runs leave orphaned branches requiring periodic git worktree prune.
  • Coordination transparency. Because subtasks are isolated, the final document's coherence depends entirely on the assembly model. Contradictions from independent subtasks pass evaluation undetected.

Summary

Git worktrees provide isolated, checkpointable, git-native working environments for concurrent subtasks without container overhead or process isolation. The pattern's implementation is forty lines of subprocess calls in orchestrator.py. Its finally block is the most important production engineering decision in the orchestrator. The pattern solves the shared-state problem completely for filesystem isolation. It does not solve coordination — and that limitation is not a defect to be engineered away. It is the boundary between two distinct architectural problems, each with its own solution.
The next chapter examines what happens when the harness escapes the boundary of a single machine entirely: when subtasks are not parallel threads in the same process, but distributed agents running on different hosts, each with different models, different memory stores, and different security boundaries — and the orchestrator must coordinate them over HTTP using the same Model Context Protocol that allows Claude Code to treat the harness as a tool.

At that scale, filesystem isolation becomes the least of your problems.
